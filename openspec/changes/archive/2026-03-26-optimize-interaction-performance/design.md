## Context

ClipMate 当前的卡顿感不是单点问题，而是几条热点路径叠加出来的结果：

- 主面板打开时，主进程会构建 `buildPanelSnapshot()`，一次性发送最多 200 条完整 `ClipItem`，其中可能包含整张图片的 base64 `content`、`link_meta` 原始 JSON 和其他重字段。
- 面板打开后，`src/renderer/src/App.vue` 会在 `clip:itemsChanged`、窗口 focus、增删改操作后执行 `refreshAfterMutation()`，重新拉取 `getClipItems(200, 0)` 和 `getSourceApps()`，导致列表、筛选、选中态和图标加载频繁整体抖动。
- 搜索使用 `LIKE` 扫描 `plain_text`、`title`、`ocr_text`、`content`、`link_meta`，既容易扫到大字段，也没有针对全文检索做专门的数据结构。
- 主窗口显示/隐藏动画在主进程中由 `setTimeout` 以 120Hz 手动驱动 `setPosition()` 和 `setOpacity()`，而面板打开链路同时还会执行剪贴板追平逻辑，进一步拉长首帧感知时延。
- 渲染层对可见卡片的应用图标加载会触发 `system:getAppIcons`，而主进程内部仍使用 `execFileSync` 调 `osascript` / `defaults` / `sips`，这类同步子进程会阻塞 Electron 主线程。

这次变更需要同时覆盖主进程、数据库、IPC 和渲染层，目标不是“微调一点点”，而是让主面板和核心交互的主观流畅度有肉眼可感的提升。

## Goals / Non-Goals

**Goals:**

- 让主面板在常见历史规模下更快显示并更早可交互，不再依赖整量数据准备完成后才能获得顺滑体验。
- 将历史列表、搜索结果和条目变更改为轻量载荷与增量同步，显著减少 IPC 传输、SQLite 扫描和渲染层重复计算。
- 优化主进程中的窗口动画、剪贴板轮询和图标/元数据后台任务，降低阻塞与空闲 CPU 占用。
- 为本次优化建立可重复的性能验证基线，确保后续可以证明“明显变快”而不是主观猜测。

**Non-Goals:**

- 不重做 ClipMate 的整体视觉风格、信息架构或功能集合。
- 不在这次变更里引入远程遥测或复杂性能监控平台。
- 不一次性重写所有 Vue 视图，只对影响性能的关键数据通路和热点交互做结构调整。

## Decisions

### 1. 引入列表摘要模型，主面板默认只消费轻量数据

新增一个面向列表展示的轻量数据模型，例如 `ClipListItemSummary`，只保留列表渲染、筛选和排序需要的字段：

- 基础元数据：`id`、`type`、`created_at`、`updated_at`、`source_app`、`source_app_name`
- 文本摘要：截断后的 `plain_text` / 标题
- 图片摘要：缩略图而不是完整 `content`
- 链接摘要：预解析后的 `linkTitle` / `linkDescription`，而不是原始 `link_meta` JSON
- 文件摘要：主文件名、数量等预处理字段

`buildPanelSnapshot()`、`getClipItems()`、`searchClipItems()` 和变更广播都改为返回摘要列表；完整 `ClipItem` 仅在预览、编辑、图片拖拽、复制原始内容等场景按需读取。

选择原因：

- 当前列表承载了超出渲染所需的数据量，尤其是图片 base64，会放大 SQLite 读取、IPC 序列化和 Vue diff 成本。
- 让列表与详情分层后，绝大多数交互都不再依赖完整对象。

备选方案：

- 继续沿用 `ClipItem` 全量对象，只做局部缓存：实现最省事，但无法根治大对象跨进程和整量重算问题。
- 仅在渲染层裁剪字段：不能减少主进程查询和 IPC 载荷，收益有限。

### 2. 用增量同步替代“变更后全量刷新”，并为搜索建立专门的数据路径

面板打开、剪贴板捕获、条目编辑、OCR 回填、链接元数据补全等事件改为发出结构化 mutation 事件，例如：

- `history:upsertSummary`
- `history:delete`
- `history:reset`
- `history:sourceAppsChanged`

渲染层维护本地 summary store，只更新受影响的条目和派生状态，不再在 `clip:itemsChanged` 后重新调用 `getClipItems(200, 0)`。

文本搜索改为使用 SQLite FTS5 虚拟表或等效全文检索结构，只索引 `plain_text`、`title`、`ocr_text` 和预处理后的链接文本字段，并通过 rowid / id 回表获取 summary 投影。类型、来源应用和日期过滤继续走普通索引条件。若当前环境无法创建 FTS 表，则退化为只搜索文本相关字段的轻量 `LIKE` 查询，明确避开图片 base64 `content`。

选择原因：

- 当前每次变化都整量重拉历史和来源应用统计，是最直观的卡顿来源之一。
- 全文搜索是高频交互，继续扫 `content` 和 `link_meta` 原始字段很难获得明显改观。

备选方案：

- 保留当前广播，只缩短 debounce：不能减少全量刷新成本。
- 只做前端本地搜索：对 200 条以内可能尚可，但无法覆盖未来数据量增长，也无法利用 OCR / 链接补全后的持久化文本。

### 3. 面板首屏显示与剪贴板追平解耦，窗口动画改为原生或渲染层驱动

主面板打开流程改成两段：

1. 立即显示窗口外壳和最近一次可用的轻量 snapshot，让键盘、滚动和选择可以尽快可用。
2. 在后台继续执行剪贴板追平；若捕获到新条目，则通过增量事件插入，而不是阻塞首屏。

主窗口显示/隐藏不再使用主进程里 120Hz 的 JS 定时器手工驱动位置与透明度，改为：

- 优先使用 macOS 原生窗口动画能力；或
- 保持窗口位置切换简单、把视觉过渡交给渲染层 CSS transform/opacity

选择原因：

- 用户对“是否立即看到面板并能操作”比对“动画是否完全同步追平数据”更敏感。
- 主线程定时动画与同步 IPC/子进程任务叠加时，很容易导致丢帧。

备选方案：

- 继续保留现有动画，只把刷新频率从 120Hz 改到 60Hz：可以缓解，但不能从根本上减少主线程占用。
- 打开面板前继续等待剪贴板完全稳定：会持续放大首帧时延。

### 4. 后台任务统一改为异步队列和按需调度

对会命中主线程的后台工作做统一调度改造：

- 剪贴板 watcher 改为自适应轮询：空闲时降低频率，捕获到变化、执行粘贴或显示面板前后进入短暂高频期。
- 应用图标加载从同步子进程改为异步队列，限制并发，只为当前可见卡片请求，结果内存 + 磁盘缓存复用。
- OCR / 链接元数据补全只广播受影响条目的 summary patch，不触发全量历史重置。

选择原因：

- 当前 `execFileSync` 和固定轮询都在持续占用主线程预算。
- 即使单次耗时不大，只要落在用户操作同时段，就会表现为“卡一下”。

备选方案：

- 仅增加缓存，不改同步调用：首次命中仍然会阻塞。
- 完全关闭应用图标或后台补全：能换性能，但会损失明显功能价值，不符合这次优化目标。

### 5. 为本次优化建立可重复的基线与预算

实现前后都记录同一组关键指标：

- 面板从触发到首屏可见的时延
- 面板从触发到首个可交互状态的时延
- 输入搜索关键字到结果稳定的时延
- 连续滚动 200 条混合历史时的掉帧感知和 CPU 峰值
- 空闲状态下主进程 CPU 占用

这些指标先通过开发环境脚本日志、Electron performance marks 和手工 smoke profiling 采集，作为回归基线写入验证结果。

选择原因：

- 没有基线，就无法判断是否达到“明显提高”。
- 这组指标覆盖了用户最容易感知卡顿的路径。

备选方案：

- 仅凭主观操作体感判断：风险过高，后续也无法防回归。

## Risks / Trade-offs

- [摘要模型与完整对象分层后，列表字段可能不够用] → 先围绕现有卡片实际展示字段设计 summary，并保留按 id 拉取完整详情的兜底路径。
- [FTS5 初始化或迁移失败导致搜索回退] → 启动时检测并记录建表结果，失败时回退到轻量 `LIKE` 查询，确保功能可用但性能目标标记为未达成。
- [增量同步实现不当可能造成列表状态与数据库不一致] → 保留 `history:reset` 全量对齐事件作为兜底，并在关键变更后增加一致性 smoke test。
- [异步图标加载和后台补全可能带来“内容逐步出现”的观感] → 优先保证交互流畅，把图标和次级元数据定义为渐进增强内容。
- [窗口动画策略调整后视觉感受与现有实现不同] → 以“更稳、更快可交互”为优先目标，通过 smoke test 校验主观体验而不是死守当前动画轨迹。

## Migration Plan

1. 为历史列表引入 summary 查询与 IPC 类型，保持详情读取接口兼容。
2. 增加搜索索引或 FTS 支撑结构，并在应用启动时做安全初始化与回填。
3. 将面板 snapshot 和条目变更广播改为增量协议，同时调整渲染层 store 与热点 watcher。
4. 替换窗口动画和后台任务调度，完成主线程热点削峰。
5. 记录优化前后的性能基线并完成关键路径回归。

回滚策略：

- 若 summary / 增量协议引入严重显示问题，可先回退到原有全量查询与广播路径。
- 若 FTS 路径在个别环境异常，可保留 legacy 查询 fallback 而不影响数据库主表数据。
- 这次变更不做破坏性数据删除；即使回滚，`clip_items` 主表数据仍可继续使用。

## Open Questions

- 是否要在本次变更里顺带把 `App.vue` 拆成 composables / 子组件，以降低后续维护成本；这对性能有帮助，但不是本 change 成功的必要条件。
- 性能验证的参考机器是否统一为当前开发机；如果需要更严格的预算，后续可能还要补一套固定测试数据集。

## Context

ClipMate 是一个以 macOS 菜单栏为主入口的 Electron 应用。当前 `src/main/tray.ts` 在运行时用内联 SVG 生成 `nativeImage`，随后直接作为 `Tray` 图标并标记为 template image。该方案没有独立资源文件、没有路径解析层，也没有对打包产物是否携带图标资源做约束。结果是当 Electron、macOS 状态栏渲染行为或打包路径与预期不一致时，应用虽然成功创建了 `Tray`，但菜单栏没有可见图标。

这个问题是跨模块修复：既涉及主进程托盘图标创建，也涉及图标资源组织、打包配置以及开发态/打包态验证方式。

## Goals / Non-Goals

**Goals:**

- 让 ClipMate 在 macOS 开发态启动时稳定显示菜单栏图标。
- 让 ClipMate 在打包后的 `.app` / DMG 运行时稳定显示菜单栏图标。
- 保留当前菜单栏交互能力：左键呼出面板、右键显示快捷菜单、暂停状态有视觉差异。
- 明确图标资源的来源、路径解析方式和缺失时的回退行为，降低回归风险。

**Non-Goals:**

- 不重做 ClipMate 的品牌视觉或整套应用图标系统。
- 不改变菜单栏菜单项、主面板窗口行为或快捷键逻辑。
- 不在这次变更里引入完整自动化 UI 测试框架。

## Decisions

### 1. 使用文件型模板图标资源替代运行时内联 SVG

将菜单栏图标切换为仓库内可追踪的模板图标资源，至少提供正常态与暂停态两个版本，并包含适合 Retina 的尺寸。主进程通过 `nativeImage.createFromPath` 加载这些资源，再调用 `setTemplateImage(true)` 保持浅色/深色模式自适应。

选择原因：

- 文件型资源更符合 macOS 状态栏图标的常见用法，渲染结果更可预测。
- 资源可以被打包、校验和人工检查，不再依赖运行时字符串生成的隐式行为。
- 正常态与暂停态可以明确区分，便于后续设计微调。

备选方案：

- 继续使用内联 SVG Data URL：实现最省事，但当前已经暴露出可见性问题，且不易定位是 SVG 解码、模板标记还是平台渲染差异导致。
- 直接复用应用 `.icns` 图标：不适合菜单栏模板图标场景，视觉上也不满足单色状态栏图标要求。

### 2. 抽出统一的托盘图标路径解析与回退逻辑

在主进程托盘模块中增加一个小型图标加载层，负责：

- 根据 `app.isPackaged` 选择打包资源路径或仓库内开发资源路径。
- 为正常态 / 暂停态返回对应的 `NativeImage`。
- 当首选资源不可读或生成空图像时，记录日志并回退到一个最小可见图标，避免再次出现“Tray 存在但没有可见图标”的失败模式。

选择原因：

- 资源路径问题在 dev 与 packaged 之间天然不同，需要集中处理。
- 回退逻辑让问题从“功能不可用”降级为“视觉降级但仍可操作”，更符合主流程恢复优先级。

备选方案：

- 在 `createTrayIcon` 内联写死多个相对路径：短期可用，但后续维护和排障成本更高。
- 完全不做回退：实现更简单，但会把任何资源问题直接暴露为用户不可用。

### 3. 把资源携带和启动验证纳入变更范围

图标资源需要被明确纳入打包配置，例如通过 `extraResources` 或保留在打包后可读取的位置，并在实现完成后进行三类验证：

- 开发态启动，确认菜单栏图标可见。
- 打包后运行 `.app`，确认菜单栏图标可见。
- 暂停/恢复切换，确认图标状态能变化且交互未回退。

选择原因：

- 当前问题同时出现在 dev 和 packaged，单一环境验证不足以覆盖风险。
- 资源如果没有被 builder 明确携带，打包态问题会持续复发。

备选方案：

- 只修代码不改验证：无法证明问题真正关闭。
- 只做打包态验证：无法覆盖开发体验回归。

## Risks / Trade-offs

- [新增图标资源后仍不符合 macOS 菜单栏显示规范] → 采用单色模板图并准备 Retina 尺寸，在开发态和打包态都做实机检查。
- [打包路径与开发路径不一致导致加载失败] → 统一通过单一 helper 解析路径，并在打包产物中显式校验资源存在。
- [回退图标掩盖真实问题，导致缺资源不易发现] → 记录明确日志，并把资源存在性检查加入验证步骤。
- [暂停态与正常态图标差异过弱] → 先保证可见性和可辨识性，视觉微调保留为后续可选优化。

## Migration Plan

1. 新增菜单栏模板图标资源并接入主进程加载逻辑。
2. 更新打包配置，确保资源在产物中可读取。
3. 在本地完成 `npm run typecheck`、开发态启动检查和打包态运行检查。
4. 若上线后发现特定机器仍有显示异常，可回滚到上一个稳定版本；代码层面保留回退图标以降低完全不可用风险。

## Open Questions

- 是否需要顺带修正应用 bundle icon（当前构建产物显示为 Electron 默认图标）；这不阻塞菜单栏图标修复，暂不纳入本 change。
